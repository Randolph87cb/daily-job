## 目标

1. 判断 `atcoder-statement-bilingual-pdf` 是否适合直接做成每周六 20:00 自动执行。
2. 判断 `atcoder-editorial-workflow` 是否适合同一时段自动执行。
3. 给出更稳妥的自动化拆分方式与落地建议。

## 关键事实

- `run_atcoder_pipeline.py` 需要显式传入 `contest id`；缺失时会直接退出。
- `atcoder-statement-bilingual-pdf` 默认依赖 AtCoder 登录态；如果会话失效，流程会自动打开可见浏览器，要求人工完成登录和 Cloudflare 校验。
- `export-editorials.ps1` 需要显式传入 `EditorialDir`；目录中没有 `*.editorial.md` 时会直接报错。
- `atcoder-editorial-workflow` 的定位更偏向“正文已准备好之后的统一导出与验证”，不适合在没有现成题解 Markdown 的情况下盲跑。

## 当前结论

- 这两个 skill 都不适合原样拼成一个“每周六 20:00 固定盲跑”的单段流程。
- `atcoder-statement-bilingual-pdf` 可以成为自动化链路的一部分，但前提是：
  - 先有稳定的 contest 发现逻辑；
  - 登录态长期有效；
  - 遇到登录失效时允许失败退出，而不是指望定时任务内交互恢复。
- `atcoder-editorial-workflow` 更适合作为第二段或独立段：
  - 只在 `editorials/` 已存在目标 Markdown 时执行；
  - 最好由“题解正文已更新”或“手工确认可导出”触发，而不是固定每周必跑。

## 建议方案

### 方案 A：拆成两段

- 周六 20:00 只跑“题面链路预检查 / 正式抓取”：
  - 自动确定 contest id；
  - 检查 `.env`、登录态、输出目录；
  - 条件满足时再调用 `run_atcoder_pipeline.py`。
- 题解导出改成独立触发：
  - 条件是 `atcoder-output/<contest>/editorials/` 下已存在 `*.editorial.md`；
  - 满足条件后再调用 `export-editorials.ps1`。

### 方案 B：20:00 只做预检查

- 如果用户坚持固定在周六 20:00 运行，建议这个时刻只做：
  - contest 识别；
  - 登录态检查；
  - 目录与依赖检查；
  - 记录待处理 contest。
- 真正的题面抓取与题解导出放到后续单独任务，避免把“交互登录”“题解未完成”“目录为空”混进同一个定时任务。

## 落地建议

- 新增一个项目级包装脚本，例如：
  - `scripts/run-weekly-atcoder-pipeline.ps1`
- 脚本负责：
  - 解析本周目标 contest；
  - 检查 `.env`、`python`、`node`、`md2pdf`；
  - 检查 AtCoder 会话是否可用；
  - 决定是否调用题面 skill 对应脚本；
  - 决定是否调用题解导出脚本；
  - 统一写日志与退出码。
- 定时自动化不要直接“调用 skill 文档”，而是调用这个包装脚本。

## 后续事项

- 如果后续确定要落地自动化，优先补：
  - contest 自动识别规则；
  - 登录失效时的失败策略；
  - 题解导出触发条件；
  - 自动化日志落点与失败通知方式。

## 后续补充：服务器场景

### 结论

- 放到服务器上以后，更不适合把这两个 skill 原样直接做成“每周六 20:00 全自动一把跑完”。
- 服务器场景比本机更严格，核心限制不是脚本能不能启动，而是：
  - 失去桌面交互后，登录恢复路径会中断；
  - 题解导出仍然缺少稳定的自动输入源；
  - 定时任务失败后需要明确日志和告警，否则很难及时发现。

### 影响判断

- `atcoder-statement-bilingual-pdf`：
  - 只有在 AtCoder 登录态长期有效、服务器能运行现有依赖、并且允许失败退出时，才适合放到服务器定时跑。
  - 一旦会话失效，当前 skill 约定的“自动打开可见浏览器并人工登录”在无桌面服务器上基本不可行。
- `atcoder-editorial-workflow`：
  - 只适合做“已有题解 Markdown 时的批处理导出”。
  - 如果服务器上没有一套稳定的题解正文生成或同步机制，固定时间跑它大概率只是报“目录为空”或没有新内容可导。

### 更稳妥的服务器方案

- 服务器上保留“无交互、可失败、可重跑”的部分：
  - contest 识别；
  - 环境检查；
  - 题面抓取与翻译；
  - 已存在题解时的批量导出。
- 登录相关操作改成“人工维护会话”：
  - 在本机完成登录；
  - 把 `browser-session-manager` 使用的会话资料同步到服务器；
  - 服务器定时任务只消费现成会话，不负责交互补救。
- 对服务器任务增加：
  - 结构化日志；
  - 明确退出码；
  - 失败告警；
  - “未发现本周 contest / 登录失效 / editorials 缺失”三类可区分状态。

### 实施建议

- 不要让服务器直接调两个 skill 的说明流程，改为调一个项目包装脚本。
- 包装脚本里拆成三段：
  - `precheck`：识别 contest、检查依赖、检查会话；
  - `statement`：条件满足才跑题面 pipeline；
  - `editorial-export`：仅当目标 `editorials/` 下已有 `*.editorial.md` 时才导出。
- 如果后续要长期服务器化，最好再补一个“登录态巡检 / 过期提醒”小脚本，而不是等周六 20:00 正式任务失败才发现。

## 后续补充：是否打包成给他人使用的通用 skill

### 当前判断

- 可以打包给别人用，但不建议把现有多个 skill 粗暴合并成一个超大 skill。
- 更合适的形态是：
  - 一个项目级“编排层” skill 或 plugin；
  - 下挂现有 `atcoder-statement-bilingual-pdf`、`atcoder-editorial-workflow` 等子 skill；
  - 再配统一入口脚本、部署说明和自动化示例。

### 原因

- 现有重复劳动的稳定部分主要在：
  - 环境检查；
  - contest 识别；
  - 会话准备；
  - 调用顺序；
  - 失败处理；
  - 日志与导出落点。
- 这些内容更像“编排和交付包装”，不应该全部塞进单个 `SKILL.md`。
- 题面抓取和题解导出本身仍然是两个相对独立的子能力，保留分层更利于维护和替换。

### 推荐落点

- 如果目标是“这个项目里的 AtCoder 流程让别人可复用”，优先做：
  - 项目级 plugin；或
  - 项目级 orchestration skill。
- 如果目标是“未来很多非 AtCoder 项目都复用同一套定时抓取/翻译/导出编排”，才考虑再抽一层全局 skill。

## 后续补充：对方没有 Codex 且不希望额外依赖

### 当前判断

- 这种场景下，不应该再以 skill、plugin 或 Codex 工作流作为最终交付物。
- 更合适的交付形态是“独立可执行程序”或“自带运行时的脚本包”。
- 如果严格要求“不要额外依赖”，就不能要求对方单独安装：
  - Codex
  - Python
  - Node
  - `md2pdf`
  - Playwright

### 推荐方案

- 优先方案：打成一个面向最终用户的 Windows 包装程序。
  - 入口形态可以是：
    - `run-atcoder-delivery.bat`
    - `run-atcoder-delivery.exe`
  - 包内自带：
    - Python 运行时或已冻结的 Python 可执行文件；
    - Node 运行时或把必须的导出逻辑改写到同一运行时；
    - 所需脚本、模板、词表和默认配置；
    - 必要时附带 Chromium 或改为纯 HTTP 抓取优先。

### 设计原则

- 对别人暴露的应该只有：
  - 一个启动入口；
  - 一个配置文件；
  - 一个输出目录；
  - 一份简短使用说明。
- Codex skill 只保留给你自己维护流程，不作为对外交付边界。

### 现实约束

- “零依赖”通常不等于“零体积”。
- 如果要真正做到别人双击即用，依赖不是消失了，而是被你一起打包进发布物里。
- AtCoder 登录态、OpenAI API Key、网络连通性仍然是外部前提，无法靠打包完全消除。
