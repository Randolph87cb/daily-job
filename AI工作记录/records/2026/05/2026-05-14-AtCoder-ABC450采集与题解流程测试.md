## 复盘

### 这次真正的目标

- 目标不是单独“翻译题面”，而是完整跑通一次 `abc450` 的交付链路：
  - 题面采集
  - 中英对照 Markdown 与 PDF
  - 题解撰写
  - 题解导出
  - 流程耗时观察与可落地优化

### 哪些信息一开始就该给

- 如果用户关心“流程流畅度和性能”，应在一开始明确性能统计口径：
  - 只看题面采集与翻译
  - 只看题解生成
  - 还是看完整交付链路
- 如果任务允许 subagent 写文件但不允许其独立提交，应该在委派 prompt 里一开始明确禁止 Git 操作，而不是在中途补拦截。
- 如果用户希望后续固定输出分阶段耗时，最好在任务开始时直接把它列为验收项，而不是跑完后再补问“题解生成时间呢”。

### 哪些返工来自需求不清、约束遗漏或验收标准不完整

- “全流程时间”口径没有在一开始定义，导致首次汇报默认按题面 pipeline 口径回答，随后又补充区分了题解导出和题解生成时间。
- 委派规则虽然要求主线程负责控制平面，但这次给子代理的 prompt 没有显式写“禁止 Git”，导致其中一个子代理提前提交并推送，虽然没有破坏结果，但增加了提交整理成本。
- 题面 PDF 导出缓存清理不是最初的明确验收项，是在执行过程中发现 `zh-CN/.tmp/md2pdf/` 大量残留后才补进脚本的。

### 下次怎么做会更好

- 对同类 AtCoder 任务，开始前固定三段计时口径：
  - `题面链路`：采集、翻译、题面 PDF
  - `题解生成`：题解 Markdown 产出
  - `题解后处理`：编译校验、样例校验、导出 PDF
- 委派写题解时，把“禁止 Git 操作”写成固定模板句，避免再次出现子代理抢先提交。
- 把题面 pipeline 的阶段计时视为默认产物，后续同类任务直接复用，不再临时补测。
- 对 PDF 相关脚本，把“导出后清理 `.tmp`”视为固定要求；这次已经在题面脚本里落地，题解脚本现状也已满足清理要求。

### 是否值得模板化、脚本化或 skill 化

- 值得继续脚本化的，是“按比赛输出分阶段耗时”的能力；这次已经先在题面 pipeline 落地，后续可以考虑把题解导出和题解验证也统一记时。
- 值得模板化的，是委派题解任务的 prompt，尤其是：
  - 文件 ownership
  - 禁止 Git
  - 验证要求
  - 汇报格式
- 目前还不需要单独新增一个新 skill；现有 `atcoder-statement-bilingual-pdf`、`atcoder-editorial-workflow` 和 `delegation-orchestrator` 已经够用，更适合先补模板和默认约束。
