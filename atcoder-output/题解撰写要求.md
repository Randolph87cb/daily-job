# AtCoder 题解撰写要求

本文档用于约束当前项目中的 AtCoder 题解写法，适用于放在 `atcoder-output/<contest>/editorials/` 下的 Markdown 题解文件。

## 1. 文件落点与命名

- 题解文件放在对应比赛目录下的 `editorials/` 子目录中。
- 单题文件命名统一为：
  - `<contest>_<problem>.editorial.md`
- 例如：
  - `abc457_a.editorial.md`
  - `abc457_f.editorial.md`

## 2. 题解固定结构

每篇题解必须包含以下一级或二级结构，顺序保持一致：

1. `# <比赛> <题号> - <题名> 题解`
2. `## 题目信息`
3. `## 题意概括`
4. `## 解题思路`
5. `## 正确性说明`
6. `## 复杂度`
7. `## 参考实现（C++，遵守代码规范）`

其中：

- `题目信息` 中至少包含：
  - 原题链接
  - 题面对照链接
  - 分值
- `题意概括` 要用简洁中文复述题意，不要直接整段照抄题面。
- `解题思路` 要讲清楚关键观察、状态含义、转移或贪心依据。
- `正确性说明` 不能只写“显然正确”，要说明为什么算法一定得到正确答案。
- `复杂度` 必须同时写时间复杂度和空间复杂度。

## 3. Markdown 格式要求

- 使用标准 Markdown 标题、列表、代码块。
- 代码必须放在带语言标记的 fenced code block 中，例如：

```cpp
// C++ code
```

- 列表前后保持空行，避免渲染器把正文和列表粘连。
- 不使用纯 LaTeX 块公式，不写 `$$...$$` 大公式块，优先使用行内公式。
- 普通说明文字保持可直接阅读，不要堆过多符号。

## 4. 公式写法要求

- 只要涉及数学表达、下标、复杂度、转移式、下标位置公式，统一使用 `$...$` 包裹。
- 常见可接受写法示例：
  - `$A_i$`
  - `$A_{i,j}$`
  - `$B_K$`
  - `$O(N)$`
  - `$O\left(\sum L_i\right)$`
  - `$pos = (K - 1) \bmod L_i + 1$`
- 不要把明显属于公式的内容强行改写成难读的纯文本。
- 但也不要把整段普通中文叙述都塞进公式中。

## 5. 代码语言与规范

- 参考实现统一使用 `C++`。
- 默认遵守项目根目录下的 [代码规范.md](../代码规范.md)。
- 当前必须遵守的关键点：
  - 统一使用 `#include<bits/stdc++.h>`
  - 使用 `using namespace std;`
  - 主函数写成 `int main(){`
  - 缩进统一 `4` 个空格
  - `if`、`else`、`for`、`while` 即使只有一行也写花括号
  - 运算符两侧保留一个空格
  - 关键逻辑前写简短教学注释
  - 优先选择更容易讲解、变量含义更直观的写法

## 6. 代码内容要求

- 代码必须与题解中的思路一致，不能题解讲一种、代码写另一种。
- 变量命名尽量贴近题面含义，例如：
  - `a[i]`
  - `l[i]`
  - `c[i]`
  - `ans`
  - `cnt`
- 当题目天然适合数组或 `vector` 表达时，优先保持下标含义清晰。
- 如果题目需要模运算、二分、前缀和、DP、图论等逻辑，注释要能对应到题解正文。

## 7. 链接要求

- 原题链接使用 AtCoder 对应题目的正式链接。
- 题面对照链接优先链接到同比赛目录下现有的 `zh-CN` Markdown。
- 相对路径要能在当前目录结构下直接打开。

## 8. 提交前自查

提交前至少检查以下内容：

1. 标题、章节层级是否统一。
2. 原题链接和题面对照链接是否可用。
3. 公式是否用 `$...$` 包裹。
4. 是否误用了 `$$...$$` 或大段 LaTeX。
5. 代码块是否标注为 `cpp`。
6. 代码是否符合 `代码规范.md`。
7. 复杂度是否完整。
8. Markdown 在普通查看器中是否仍可顺畅阅读。

## 9. 当前适用范围

- 当前文档适用于本项目里 AtCoder 题解的现行写法。
- 如果后续用户对格式、公式、代码风格、目录结构提出新的明确要求，应先更新本文档，再继续批量补题解。
